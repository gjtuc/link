## 0. 이 글을 읽기 전에 — 한 문장 요약

**Link에 “뼈대(Spine)” 읽기 기능을 붙입니다.**  
PDF를 올리면 AI가 글을 잘게 쪼개고(원인과 결과), 검증된 인과를 화살표로 이어 줍니다.  
사용자는 **왼쪽 뼈대 목록**으로 큰 흐름을 고르고, **화살표에 마우스**를 올려 한 칸씩 인과를 읽고, **클릭**하면 옆 패널에서 더 깊게 봅니다.  
**저자가 쓴 목차 순서가 아니라, 인과 메커니즘 순서**로 읽게 하는 것이 목표입니다.

**지금 단계:** 설계만 끝냈고, **아직 화면에 구현하지 않았습니다.**

## 1. 전체 약어 사전 (여기서 먼저 읽어도 됩니다)

아래 표에 나오는 **모든 약어**를 풀어 씁니다. 본문에서 다시 나와도 같은 뜻입니다.

### 1-1. 제품·단계 이름

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **Link** | 링크 | 이 PC에서 돌아가는 **인과 그래프 웹 UI** (`http://127.0.0.1:8765/`). 논문·보고서 PDF를 분석해 원인과 결과 지도를 보여 줌. |
| **SPINE** / **0-SPINE** | 스파인 / 제로-스파인 | **읽기 UX** 전체 프로젝트 이름. “뼈대 따라 읽기” 레이어. |
| **Spine** (소문자 spine) | 뼈대 | 그래프 안에서 **인과가 2칸 이상** 이어진 **읽기용 경로** 하나. 목록에 `뼈대 1 · A → B`처럼 나옴. |
| **Branch-SPINE** | 브랜치-스파인 | SPINE 기능을 코드에 넣는 **개발 브랜치(작업 줄기)** 이름. |
| **STAGE 0** | 스테이지 0 | Link 제품의 **첫 큰 단계** 설계 묶음. |
| **STAGE 0-1** | 스테이지 0-1 | SPINE 이전의 **원인과 결과 지도** 기본 정의 문서. |
| **STAGE 0-SPINE-1** | 스테이지 0-스파인-1 | 철학·인터랙션 계약 문서 (`STAGE-0-SPINE-philosophy.md`). |
| **STAGE 0-SPINE-2** | 스테이지 0-스파인-2 | 사용자 **시나리오** 문서 (`STAGE-0-SPINE-2-scenarios.md`). |
| **STAGE 0-SPINE-3** | 스테이지 0-스파인-3 | **합격 기준(AC)** 문서 (`STAGE-0-SPINE-3-acceptance.md`). |
| **Branch-0** | 브랜치 0 | SPINE 작업 전에 이미 있던 **기본 기능**. SPINE을 넣어도 이게 깨지면 안 됨(회귀). |
| **Branch-2b** | 브랜치 2b | 여러 PDF를 한 번에 올릴 때 **교차 연결(BRIDGE)** 을 만드는 기존 작업. |

### 1-2. 그래프·데이터 용어

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **원인과 결과** | 인과 | **원인**과 **결과**를 잘게 쪼갠 조각(crumb). Link의 기본 단위. |
| **fact** / **팩트** | 팩트 | 그래프의 **노드(점)** 하나. “무슨 일이 관측·서술되었는가” 조각. |
| **edge** | 엣지 / 화살표 | 그래프의 **선**. 두 팩트 사이 관계. |
| **node** | 노드 | 그래프의 **점** = 팩트. |
| **CAUSES** | 코즈 / 검증 인과 | **검증된 인과** 화살표. “A가 B를 일으킨다”고 시스템이 검증한 연결. hover 시 `[검증 인과]` 태그. |
| **BRIDGE** | 브리지 / 교차 연결 | **서로 다른 PDF**에서 **같은 주제(subject)** 가 맞을 때 잇는 **보라색** 선. **인과가 검증된 것은 아님**. hover 시 `[교차 연결]`. |
| **subject** | 서브젝트 / 주제 | 팩트가 말하는 **핵심 대상** (예: “Ni 촉매”, “반응 온도”). BRIDGE는 subject가 같을 때만 생김. |
| **Gap** | 갭 | 인과 **끝**에서 “여기서 더 이어지지 않음”을 표시하는 **품질 신호**. 실패가 아님. |
| **Weak** | 위크 | 약한 연결·불확실 구간 **품질 신호**. |
| **Strong chain** | 스트롱 체인 | 코드에 있는 **강한 인과 줄기 후보** (`find_strong_chains`). Spine의 **재료**이지만, Spine과 **완전히 같은 말은 아님**. |
| **skeleton** | 스켈레톤 | 기존 UI의 **품질 요약·참고 목록** (Gap/Strong/Weak, outline). Spine이 **대체하는 것이 아니라** 옆에 남음. |
| **force graph** | 포스 그래프 | 점·선이 힘으로 퍼져 보이는 **탐색용 그래프**. SPINE에서는 **배경**; 뼈대 밖은 흐리게(dim). |
| **dim** | 딤 | 선택한 뼈대 **밖** 노드·선을 **흐리게** 표시. |
| **DAG** | 다그 | **방향 있는 비순환 그래프**. 화살표가 한 방향으로만 가고, 루프 없음. 가지가 갈라져도 됨. |
| **main path** | 메인 패스 / 주 경로 | 가지가 있는 뼈대에서 **가장 긴 검증 인과(CAUSES) 줄기**. 목록 이름·따라가기의 기본. |
| **merge** / **병합** | 병합 | 여러 PDF 분석 결과를 **한 그래프**로 합침. |
| **chunk** / **청크** | 청크 | PDF를 잘라 넣는 **텍스트 덩어리** 하나. |
| **fixture** | 픽스처 | 테스트용 **미리 만들어 둔 JSON 샘플** 파일. |
| **API** | API | 브라우저가 서버에 “분석 결과 줘”라고 요청할 때 쓰는 **데이터 형식**. |
| **JSON** | 제이슨 | `{ "이름": "값" }` 형태의 **데이터 파일 형식**. |
| **Neo4j** | 네오포제이 | 그래프 DB 이름. **1차 SPINE에서는 DB 구조를 바꾸지 않음** — 세션 JSON 우선. |

### 1-3. 필드·문장 ID (R4, R6 등)

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **R4** | 알포 | 필드 이름 `link_sentence`. **화살표 hover 두 번째 줄**에 나오는 **짧은 인과 한 문장** (~120자). |
| **R6** | 알식스 | 필드 이름 `link_mechanism`. **화살표 click 후** 옆 패널에 나오는 **인과 메커니즘 설명** (CAUSES만). |
| **link_sentence** | 링크 센텐스 | = R4. 코드·JSON에서 쓰는 **정식 필드 이름**. |
| **link_mechanism** | 링크 메커니즘 | = R6. 코드·JSON에서 쓰는 **정식 필드 이름**. |
| **link_rationale** | 링크 레이셔널 | 화살표 하나에 대한 **R4+R6+종류(CAUSES/BRIDGE)** 묶음 데이터. |
| **T4** | 티포 | R4 문장을 만들 때 쓰는 **문장 틀(템플릿)** 규칙 번호. |
| **state_change** | 스테이트 체인지 | 팩트에 붙는 “무엇이 어떻게 변했는가” 정보. R4 문장에 반영. |
| **5b-B** | 오비-비 | 검색 맥락 규칙 ID. **기본은 좁게**, 체크 시 넓게. 아래 §4-3 참고. |
| **7b** | 세븐비 | BRIDGE hover 문장 규칙. **인과라고 말하지 않고** “교차 연결”만 설명. |

### 1-4. 신뢰 층 (E / V / H / B / G / S) — δ

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **δ** (델타) | 델타 | **출처·신뢰 층**을 숨기지 말라는 **제품 원칙**. “이 말은 어디서 왔나?”를 UI에 표시. |

| **S** | 에스 / Searched | **검색·질문 응답** (Tavily 등). R6 아래 대화 블록. |

### 1-6. 작업 단위 μ, ω, AC

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **μ** (뮤) | 뮤 | **가장 작은 구현 작업 한 덩어리** ID. 예: `μ-SPINE-01` = R4/R6 데이터 만들기. |
| **μ-SPINE-00** | 뮤-스파인-00 | **설계만** 하는 단계. 지금 읽는 spec. 코드 거의 없음. |
| **μ-SPINE-01** | 뮤-스파인-01 | `link_sentence` / `link_mechanism` **생성 코드** 첫 착수. |
| **μ-SPINE-02** | 뮤-스파인-02 | **뼈대 목록을 계산**하는 `spine_index` 코드. |
| **μ-SPINE-02-API** | 뮤-스파인-02-API | 분석 결과 JSON에 `spine`, `link_rationales` **넣기**. |
| **μ-SPINE-03-UI** ~ **05-UI** | 뮤-스파인-03~05 UI | 화면: 목록·hover / 패널 / 검색 체크박스. |
| **μ-SPINE-ω** | 뮤-스파인-오메가 | **마무리 통합 테스트** — fixture A~D + MUST 합격 기준 한 번에. |
| **μ-SPINE-BUNDLE** | 뮤-스파인-번들 | **2차**: 주제 묶음(§10). |
| **μ-CHUNK-STRUCT** | 뮤-청크-스트럭트 | **별도 작업**: PDF를 **절·문단** 단위로 자르기 (§6). |
| **μ-2b** | 뮤-2b | 기존 **다중 PDF / BRIDGE** 유지보수. SPINE과 **병행**. |
| **ω** (오메가) | 오메가 | 한 **큰 사이클** 끝날 때 돌리는 **통합 검증** 묶음. |
| **AC** / **C-SPINE-xx** | 어셉턴스 크리테리아 | **합격 기준** 번호. 예: `C-SPINE-01` = CAUSES hover 2줄. |
| **C-BUNDLE-xx** | 씨-번들 | 주제 묶음 **합격 기준**. |
| **C-CHUNK-xx** | 씨-청크 | 절·문단 청크 **합격 기준**. |
| **DoD** | 디오디 | Definition of Done — **끝냈다고 말하려면** 채워야 할 조건. |
| **E2E** | 이투이 | End-to-End — **브라우저까지** 포함한 통합 테스트. |
| **UAT** | 유에이티 | 사용자가 직접 시나리오를 밟아 보는 **사용자 검수**. |
| **Worker** | 워커 | 나중에 **코드를 짜는** 역할(개발 에이전트). 감독이 spec 승인 후 주문. |
| **W1~W5** | 더블유 1~5 | Worker가 μ 끝낼 때 확인하는 **체크리스트** (테스트, fixture, 문서 헤더 등). |
| **branch_spine_unlocked** | 브랜치 스파인 언락드 | `true`가 되면 **구현 착수 허가** 플래그. 감독 승인 전에는 코드 대규모 수정 금지. |

### 1-7. 시나리오·판정 약어

| 약어 | 풀어 쓴 말 | 뜻 |
|------|------------|-----|
| **S-SPINE-A ~ J** | 에스-스파인-A~J | **사용자 시나리오** 번호. A=뼈대 따라 읽기, B=뼈대 없음, C=교차 연결… |
| **P0 / P1 / P2** | 피0 / 피1 / 피2 | 시나리오 **우선순위**. P0=1차 필수, P1=1차 권장, P2=2차. |
| **P1, P2, P4** (§2) | 피원, 피투, 피포 | **최소 PASS 신호** 이름. P1=뼈대≥1, P2=최장 자동 선택, P4=hover 2줄. |
| **PASS / FAIL / WARN** | 패스 / 페일 / 워닝 | 합격 / 불합격 / 경고(동작은 하되 주의). |
| **F1, F4, F5** | 에프1 등 | **실패 유형** 코드. F1=뼈대 0, F4=뼈대 UI 없음, F5=BRIDGE를 인과처럼 표시. |
| **W2** | 더블유2 | **팩트가 적음** 경고 문구. |
| **NG-3** | 엔지-3 | **웹 검색** 쓸 때 반드시 보여 줄 **고정 안내 문구** (출처·한계). |
| **NG-SPINE-4** | 엔지-스파인-4 | 노드 hover 금지, force graph만 있고 뼈대 목록 없음 등 **하지 말 것**. |
| **MUST / SHOULD / MAY** | 머스트 / 슈드 / 메이 | 반드시 / 하면 좋음 / 있어도 됨(2차). |
| **TopicBundle** | 토픽 번들 | §10 **주제 묶음** — 비슷한 subject의 뼈대 여러 개를 **나란히** 보기. 인과 합치지 않음. |

### 1-8. 철학 재쪼개기 α ~ ζ (읽기 선택)

| 약어 | 풀어 쓴 말 | 뜻 |

## 2. 지금 Link에는 뭐가 있고, SPINE으로 뭐가 바뀌나

### 2-1. 지금 (구현 전)

| 있는 것 | 없는 것 (SPINE 목표) |
|---------|----------------------|
| PDF 올리면 팩트·화살표 그래프 | 왼쪽 **뼈대 목록** |
| CAUSES 화살표 hover에 **지연 시간** 정도만 | hover **2줄** (태그 + R4 문장) |
| skeleton Gap/Strong 요약 | **최장 뼈대 자동 선택** + 밖은 dim |
| 여러 PDF → BRIDGE 보라 선 | 뼈대 라벨에 `(교차 k)` |
| 노드에 마우스 올리면 툴팁 있을 수 있음 | 노드 hover **없음** → click만 패널 |
| — | 화살표 click → **인과 상세** + R6 + 검색 |

### 2-2. 바꾸지 않는 것

- PDF 쪼개기(Deconstruct) → 검증(Skeptic) → 그래프 만드는 **순서**
- skeleton, recompose, force graph **삭제 안 함** — 역할만 나눔
- BRIDGE는 **인과 검증 아님** — 그대로 라벨로 구분

## 3. 화면을 천천히 걸어 보기

아래는 **완성 후** 사용자가 보게 될 화면입니다. (아직 코드 없음)

```
┌──────────────────┬────────────────────────────────────┐
│  왼쪽 위          │  오른쪽 (큰 영역)                   │
│  [뼈대 목록]      │  [인과 그래프]                      │
│                  │                                    │
│  ▶ 뼈대 1 ·      │   선택한 뼈대 안 = 선명              │
│    원료 → 수율   │   나머지 = 흐림(dim)                │
│    뼈대 2 ·      │                                    │
│    … (교차 1)    │   점 = 팩트                        │
│                  │   화살표 = CAUSES 또는 BRIDGE       │
│  [가설 포함]     │                                    │
│  [교차 연결 포함]│                                    │
├──────────────────┤                                    │
│  왼쪽 아래        │                                    │
│  [팩트 상세 |     │                                    │
│   인과 상세]      │                                    │
│  (하나의 패널)    │                                    │
│                  │                                    │
│  ☐ 다른 업로드    │                                    │
│    문서도 참고    │                                    │
│  [질문 입력…]     │                                    │
└──────────────────┴────────────────────────────────────┘
```

### 3-1. 뼈대 목록 (왼쪽 위)

- 분석이 끝나면 시스템이 **읽기 좋은 인과 줄기**를 여러 개 찾습니다.
- 각 줄기 = **뼈대 하나**.
- 이름 예: `뼈대 1 · Ni 촉매 → 메탄 수율`
- 다른 논문을 잇는 구간이 있으면: `뼈대 2 · A → B (교차 1)` — **교차 1** = BRIDGE 1개 포함.
- **가장 긴 뼈대**가 처음에 **자동 선택**됩니다.
- 다른 뼈대에 마우스를 올리면 **미리보기**, 클릭하면 **고정 선택**.

**뼈대가 하나도 없을 때** (인과가 1칸만 검증된 경우):

- 메시지: `뼈대 없음 (인과 2칸 이상 필요)`
- 그래도 **점·화살표는 보임** — 분석 실패가 아님.

### 3-2. 그래프 (오른쪽)

- **선택된 뼈대** 안의 점·선은 선명.
- **밖**은 흐리게 — 한 줄기에 집중하라는 뜻.
- **노드(점):** 마우스 올려도 **아무 말 안 함**. **짧게 클릭** → 왼쪽 아래 `팩트 상세`.
- **화살표:**  
  - **마우스 올림 (0.12초 후):** 2줄 툴팁  
    - 1줄: `[검증 인과]` 또는 `[교차 연결]`  
    - 2줄: R4 짧은 문장  
  - **클릭:** 툴팁 닫힘 → `인과 상세` 패널 (R6, 검색).

### 3-3. 필터 두 개

| 필터 | 기본 | 의미 |
|------|------|------|
| `[가설 포함]` | **꺼짐** | 켜면 아직 검증 안 된 가설 단계도 뼈대에 넣을 수 있음. |
| `[교차 연결 포함]` | **켜짐** | 뼈대에 **BRIDGE(교차)** 구간을 넣을지. |

### 3-4. 옆 패널 (왼쪽 아래) — 하나만

- **패널은 1개.** 마지막으로 클릭한 것이 내용을 바꿈.
- **노드 클릭** → 제목 `팩트 상세`: subject, 출처 파일, 페이지 등.
- **화살표 클릭** → 제목 `인과 상세`: R4 다시 + R6 + 양끝 요약 + 질문 칸.
- 패널이 열려 있어도 **다른 화살표에 hover 가능**.

### 3-5. 검색·「다른 업로드 문서도 참고」 (5b-B)

- **체크 꺼짐 (기본, 좁은 맥락):** 질문할 때 **이 화살표 양끝 팩트 + R4 + R6**만 참고.
- **체크 켜짐 (넓은 맥락):** 이번에 올린 **모든 PDF**를 참고. 패널에 `교차 비교 모드` 한 줄 표시.
- 넓은 모드도 **“두 논문이 같은 인과다”라고 합치지는 않음** — 나열·대조만.
- 웹 검색 쓰면 **NG-3** 안내 문구가 반드시 보임.

### 3-6. 2차에 올 기능 (지금 설계만)

- **다음 칸 / 이전 칸** 버튼으로 주 경로 따라가기 (S-SPINE-H).
- **주제 묶음** — 같은 주제 뼈대 여러 개 묶어 보기 (S-SPINE-J).

## 4. 데이터 — 개발자가 저장할 것 (쉬운 설명)

코드는 나중에 Worker가 짜지만, **무엇을 저장하기로 했는지**만 이해하면 됩니다.

### 4-1. 화살표 한 개 설명: `LinkRationale`

| 필드 | 쉬운 뜻 |
|------|---------|
| `source_fact_id` | 화살표 **시작** 팩트 ID |
| `target_fact_id` | 화살표 **끝** 팩트 ID |
| `edge_kind` | `CAUSES` 또는 `BRIDGE` |
| `link_sentence` | **R4** — hover용 한 문장 |
| `link_mechanism` | **R6** — 패널용 메커니즘 (BRIDGE는 **빈 문자열**) |
| `locale` | R4 언어 (`ko` / `en`) — 태그는 한국어 고정 |

**R4가 없을 때:** 시스템이 A·B 정보로 **기계적으로 짧게 조합** (§1d-8). “자동 생성”이라는 말은 UI에 안 붙임 (1차).

### 4-2. 뼈대 한 줄: `SpineRecord`

| 필드 | 쉬운 뜻 |
|------|---------|
| `spine_id` | 이 분석 실행에서 뼈대 **고유 ID** |
| `index` | 화면에 보이는 번호 (1부터: 뼈대 1, 뼈대 2…) |
| `label` | `원료 → 수율` 같은 **읽기 이름** |
| `bridge_count` | 이 뼈대 안 **BRIDGE 개수** |
| `node_ids` | 뼈대에 들어가는 **모든 점** |
| `edge_ids` | 뼈대에 들어가는 **모든 화살표** (시작→끝 쌍) |
| `main_path_node_ids` | **주 경로** 위 점만 (굵게 그릴 줄기) |
| `is_branched` | 갈라짐이 있으면 `true` |

### 4-3. 분석 결과 JSON에 추가되는 부분

분석 API 응답에 대략 이런 블록이 **더 붙습니다**:

```json
{
  "spine": {
    "spines": [ ... SpineRecord 목록 ... ],
    "selected_spine_id": "자동 선택된 뼈대 ID",
    "filters": {
      "include_hypothesis": false,
      "include_bridge": true
    }
  },
  "link_rationales": [ ... 화살표별 R4/R6 ... ]
}
```

기존 `skeleton` 블록은 **그대로** 둡니다.

## 5. 뼈대를 어떻게 고르는지 (알고리즘 쉬운 버전)

Worker용 상세는 [BRANCH-SPINE-spec.md](BRANCH-SPINE-spec.md)에 있고, 요지만 적습니다.

1. 여러 PDF 결과를 **한 그래프로 합친다**.
2. **검증 인과(CAUSES)** 화살표만으로 **2칸 이상** 이어진 후보 줄기를 찾는다 (기존 `find_strong_chains`와 비슷).
3. 필요하면 **BRIDGE**로 줄기를 **이어 붙일 수 있다** — 단, **CAUSES가 최소 1칸은** 있어야 뼈대가 됨. **BRIDGE만**인 줄기는 뼈대 **아님**.
4. 갈라짐 **허용** — 나무 가지처럼. 그중 **가장 긴 CAUSES 줄** = 주 경로.
5. 뼈대 많으면 **긴 주 경로 순**으로 정렬, 화면에는 최대 20개 + “더 보기”.
6. `(교차 k)` = 그 뼈대 안 BRIDGE **k개**.

## 8. 합격 기준 — 1차에 꼭 되어야 하는 것

**1차 μ-SPINE ω에서 MUST 전부 통과:**

| AC 번호 | 쉬운 말 |
|---------|---------|
| **C-SPINE-05** | 뼈대가 있으면 **목록이 보이고**, 가장 긴 게 **선택**됨 |
| **C-SPINE-01** | CAUSES 화살표 hover = **`[검증 인과]` + R4** 두 줄 |
| **C-SPINE-09** | 뼈대 있거나, 없으면 **없음 메시지** — 둘 중 하나는 반드시 맞음 |
| **C-SPINE-02** | 노드 hover 없음, click → **팩트 상세** 패널 |
| **C-SPINE-08** | 뼈대 0일 때 **정해진 문구** 표시 |

**하면 좋은 것 (SHOULD):** R6 패널, BRIDGE 라벨 `(교차 k)`, BRIDGE hover 문장, sparse 경고 등.

| `spine_closure_a.json` | 인과 3칸 + R4 (A) |

| `spine_closure_c.json` | PDF 2개 + BRIDGE (C) |
